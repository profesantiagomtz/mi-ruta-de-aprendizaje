Sección 1 · Introducción

El lenguaje de las bases de datos

Bienvenido a tu Cuaderno Digital de Aprendizaje. En esta unidad aprenderás a comunicarte con una base de datos usando SQL, el mismo lenguaje que utilizan bancos, hospitales, empresas y sistemas escolares en todo el mundo.

¿Qué es SQL?

SQL (Structured Query Language, o Lenguaje Estructurado de Consultas) es el lenguaje estándar para trabajar con bases de datos relacionales. Con SQL puedes hacer cuatro cosas esenciales con la información:

  • Insertar datos nuevos (INSERT) — por ejemplo, dar de alta a un alumno.
  • Consultar datos existentes (SELECT) — por ejemplo, obtener la lista de un grupo.
  • Actualizar datos (UPDATE) — por ejemplo, corregir un apellido mal capturado.
  • Eliminar datos (DELETE) — por ejemplo, borrar un registro duplicado.

A estas operaciones se les conoce como CRUD (Create, Read, Update, Delete) y son la base de prácticamente cualquier sistema de información.

¿Dónde se usa SQL en la vida real?

Cada vez que interactúas con un sistema, hay una base de datos detrás respondiendo consultas SQL:

  • Cuando el banco consulta tu saldo, ejecuta un SELECT.
  • Cuando una tienda en línea registra tu pedido, ejecuta un INSERT.
  • Cuando el IMSS actualiza tu clínica asignada, ejecuta un UPDATE.
  • Cuando control escolar de tu plantel genera tu boleta, ejecuta consultas con SELECT e INNER JOIN sobre varias tablas.

Sección 2 · Repaso · RA 1.3, contenido B

Antes de usar los datos, hay que construir la casa: el DDL

En la Unidad 1 aprendiste a construir la estructura de una base de datos; en la Unidad 2 aprenderás a trabajar con sus datos. Este repaso del RA 1.3 te recuerda las herramientas de construcción, porque las vas a necesitar para entender todo lo que sigue.

SQL tiene dos familias de comandos

FamiliaSignificado¿Sobre qué actúa?Comandos
DDLLenguaje de Definición de DatosLa estructura: bases y tablasCREATE, ALTER, DROP
DMLLenguaje de Manipulación de DatosLos datos dentro de las tablasSELECT, INSERT, UPDATE, DELETE

Piénsalo así: el DDL construye los estantes del archivero; el DML guarda, busca, corrige y saca los expedientes. Esta unidad (RA 2.1) es pura DML — pero sin estantes no hay dónde guardar nada.

1 · Creación de la base de datos

Todo empieza creando el "contenedor" general y diciéndole al gestor que vamos a trabajar dentro de él:

🗄 Sistema gestor de bases de datos
-- Crear el contenedor del proyecto integrador
CREATE DATABASE SistemaAdministracionEstudiantil;

-- Indicar al gestor que trabajaremos dentro de ella
USE SistemaAdministracionEstudiantil;

2 · CREATE TABLE — crear tablas

Al crear una tabla defines sus columnas, el tipo de dato de cada una y sus reglas. Así nació la tabla que ya conoces:

🗄 SistemaAdministracionEstudiantil › alumnos
CREATE TABLE alumnos (
    matricula        VARCHAR(10) PRIMARY KEY,
    nombre           VARCHAR(40) NOT NULL,
    apellido_paterno VARCHAR(40) NOT NULL,
    grupo            VARCHAR(5),
    turno            VARCHAR(12),
    promedio         DECIMAL(4,2)
);
Elemento¿Qué significa?
VARCHAR(40)Texto de hasta 40 caracteres (nombres, grupos, turnos).
DECIMAL(4,2)Número con decimales: 4 dígitos en total, 2 después del punto (ej. 9.45).
INTNúmero entero (identificadores, parciales).
PRIMARY KEYLlave primaria: identifica de forma única a cada fila. Por eso la matrícula no puede repetirse.
NOT NULLLa columna no puede quedar vacía al capturar.

3 · ALTER TABLE — editar o actualizar tablas

Las necesidades cambian y la estructura también. ALTER TABLE modifica una tabla sin perder los datos que ya contiene:

🗄 SistemaAdministracionEstudiantil › alumnos
-- Agregar una columna nueva
ALTER TABLE alumnos ADD apellido_materno VARCHAR(40);

-- Cambiar el tipo o tamaño de una columna existente
ALTER TABLE alumnos MODIFY grupo VARCHAR(8);

-- Quitar una columna (¡y sus datos!)
ALTER TABLE alumnos DROP COLUMN apellido_materno;
No confundas ALTER con UPDATE

ALTER TABLE cambia la estructura (columnas, tipos). UPDATE — que verás en esta unidad — cambia los datos dentro de las filas. Son familias distintas: DDL vs DML.

4 · DROP TABLE — eliminar tablas

🗄 SistemaAdministracionEstudiantil
-- Eliminar una tabla completa
DROP TABLE tabla_de_pruebas;
⚠ DROP no es DELETE

DELETE borra filas (datos) y la tabla sigue existiendo. DROP TABLE elimina la tabla completa: estructura y datos, sin vuelta atrás. En un sistema en producción, un DROP equivocado es catastrófico.

5 · Relación de tablas — llaves foráneas

Aquí está el secreto que hace funcionar todo el sistema: una llave foránea (FOREIGN KEY) es una columna que "apunta" a la llave primaria de otra tabla, conectándolas. Así se creó calificaciones, que se relaciona con alumnos y con materias:

🗄 SistemaAdministracionEstudiantil › calificaciones
CREATE TABLE calificaciones (
    id_calificacion INT AUTO_INCREMENT PRIMARY KEY,
    matricula       VARCHAR(10) NOT NULL,
    id_materia      INT NOT NULL,
    calificacion    DECIMAL(4,2),
    parcial         INT,
    FOREIGN KEY (matricula)  REFERENCES alumnos(matricula),
    FOREIGN KEY (id_materia) REFERENCES materias(id_materia)
);
Conexión con la Unidad 2

Estas llaves foráneas garantizan la integridad referencial: no puede existir una calificación de una matrícula que no exista en alumnos. Y son exactamente las columnas que usarás como "puente" en los INNER JOIN ... ON de la Actividad 3. Cuando llegues ahí, recuerda este momento.

6 · Herramientas CASE para el diseño

Las herramientas CASE (Computer-Aided Software Engineering, ingeniería de software asistida por computadora) te ayudan a diseñar visualmente la base de datos antes de escribir una sola línea de DDL — y muchas generan el código por ti:

  • MySQL Workbench: dibujas el diagrama entidad-relación (EER) con tablas, llaves y relaciones, y con Forward Engineering genera automáticamente todos los CREATE TABLE.
  • dbdiagram.io (en línea, gratuito): describes las tablas en texto sencillo y produce el diagrama y el SQL.
  • draw.io / Lucidchart: para diagramar el modelo y los casos de uso del sistema en la etapa de análisis.

El flujo profesional es: diseñar en la herramienta CASE → generar el DDL → ejecutarlo en el gestor → empezar a trabajar con DML (que es justo lo que harás en este cuaderno).

Sección 3 · Conceptos fundamentales

Las siete herramientas que debes dominar

Cada concepto se explica con su sintaxis general y un ejemplo real sobre la base de datos del proyecto. Observa que en todos los ejemplos se indica la base de datos y la tabla con la que se trabaja: ese hábito te acompañará todo el semestre.

SELECT — consultar información

SELECT recupera información de una tabla. Indicas qué columnas quieres ver y de qué tabla provienen.

-- Sintaxis general
SELECT columna1, columna2
FROM nombre_tabla;

-- El asterisco (*) significa "todas las columnas"
SELECT * FROM nombre_tabla;
🗄 SistemaAdministracionEstudiantil › alumnos
-- Obtener nombre y grupo de todos los alumnos
SELECT nombre, apellido_paterno, grupo
FROM alumnos;

WHERE — filtrar resultados

WHERE establece una condición: solo se incluyen las filas que la cumplen. Sin WHERE, la instrucción afecta o muestra todas las filas de la tabla.

🗄 SistemaAdministracionEstudiantil › alumnos
-- Solo los alumnos del grupo 410
SELECT nombre, apellido_paterno, promedio
FROM alumnos
WHERE grupo = '410';

Columnas calculadas con operadores aritméticos

Los operadores aritméticos no solo sirven en el WHERE: también puedes usarlos dentro del SELECT para crear una columna calculada — una columna nueva que el gestor calcula al vuelo, sin modificar la tabla:

🗄 SistemaAdministracionEstudiantil › alumnos
-- Tercera columna calculada: el promedio en escala de 0 a 100
SELECT nombre, promedio, promedio * 10
FROM alumnos;

El resultado muestra tres columnas: para María (promedio 9.4), la calculada vale 94. La tabla alumnos no cambia: el cálculo solo existe en el resultado de la consulta.

INSERT — agregar registros

INSERT INTO agrega una fila nueva. Se indican las columnas y, en el mismo orden, los valores.

🗄 SistemaAdministracionEstudiantil › alumnos
-- Dar de alta a un alumno de nuevo ingreso
INSERT INTO alumnos (matricula, nombre, apellido_paterno, grupo, turno, promedio)
VALUES ('24100501', 'Diana', 'Cervantes', '410', 'Matutino', 0.0);

UPDATE — modificar registros

UPDATE cambia el valor de una o varias columnas en las filas que cumplan la condición del WHERE.

🗄 SistemaAdministracionEstudiantil › alumnos
-- Cambiar de turno a un alumno específico
UPDATE alumnos
SET turno = 'Vespertino'
WHERE matricula = '23100459';

DELETE — eliminar registros

DELETE FROM borra filas completas. Igual que UPDATE, depende del WHERE para saber cuáles.

🗄 SistemaAdministracionEstudiantil › alumnos
-- Eliminar el registro de un alumno dado de baja
DELETE FROM alumnos
WHERE matricula = '23100378';
⚠ Regla de oro del técnico en bases de datos

Un UPDATE o un DELETE sin WHERE modifica o borra TODAS las filas de la tabla. En un sistema real eso puede significar perder la información de cientos de alumnos en un segundo. Antes de ejecutar, verifica siempre tu condición — idealmente probándola primero con un SELECT.

INNER JOIN — combinar tablas

La información de un sistema real está repartida en varias tablas. INNER JOIN las une a través de una columna en común (normalmente una llave, como matricula), devolviendo solo las filas que coinciden en ambas.

🗄 SistemaAdministracionEstudiantil › alumnos + calificaciones
-- Nombre del alumno junto a sus calificaciones
SELECT alumnos.nombre, calificaciones.calificacion
FROM alumnos
INNER JOIN calificaciones
        ON alumnos.matricula = calificaciones.matricula;

La cláusula ON es el "puente": le dice a la base de datos qué columna conecta una tabla con la otra.

¿Y si necesito TRES tablas? Se encadenan dos INNER JOIN

Cada INNER JOIN trae una tabla nueva a la consulta, con su propio ON indicando el puente que la conecta. Observa cómo calificaciones se une con alumnos (por matricula) y con materias (por id_materia):

🗄 SistemaAdministracionEstudiantil › alumnos + materias + calificaciones
SELECT alumnos.nombre, materias.nombre_materia, calificaciones.calificacion
FROM calificaciones
INNER JOIN alumnos  ON calificaciones.matricula  = alumnos.matricula
INNER JOIN materias ON calificaciones.id_materia = materias.id_materia;

Fíjate que cada columna lleva el nombre de su tabla (alumnos.nombre): cuando hay varias tablas en juego, así el gestor sabe exactamente de cuál hablas. Este patrón de dos JOIN encadenados es el que usarás en la Actividad 3.

Disparadores (Triggers) — la base de datos reacciona sola

Un disparador es un bloque de código que la base de datos ejecuta automáticamente cuando ocurre un evento sobre una tabla. Nadie lo llama: se "dispara" solo. Es la herramienta clásica para auditorías, validaciones y cálculos automáticos.

  • Momento: BEFORE (antes de que el evento se aplique) o AFTER (después de aplicado).
  • Evento que lo activa: un INSERT, un UPDATE o un DELETE sobre la tabla indicada.
  • Acción: lo que el disparador ejecuta, una vez por cada fila afectada (FOR EACH ROW).
🗄 SistemaAdministracionEstudiantil › calificaciones
-- Cada calificación nueva queda registrada en una bitácora, automáticamente
CREATE TRIGGER tr_calificacion_registrada
AFTER INSERT ON calificaciones
FOR EACH ROW
INSERT INTO bitacora_capturas (id_materia, fecha_evento)
VALUES (NEW.id_materia, NOW());
OLD y NEW: las dos "fotografías" de la fila

Dentro de un disparador, NEW es la fila como queda después del evento (existe en INSERT y UPDATE) y OLD es la fila como estaba antes (existe en UPDATE y DELETE). Así puedes registrar qué cambió y de qué registro.

Análisis con el método de 4 pasos

  1. ¿Qué información se solicita? Nombre, apellido y promedio. → Columnas nombre, apellido_paterno, promedio.
  2. ¿En qué tabla se encuentra? Todos esos datos viven en la tabla alumnos.
  3. ¿Una o varias tablas? Una sola: todo lo solicitado está en alumnos.
  4. ¿Requiere JOIN? No, porque no necesitamos combinar información de otra tabla.

Sección 8 · Actividad práctica 4

Disparadores · Auditoría automática de cambios

CREATE TRIGGERAFTER / BEFOREOLD y NEW20 puntos

📋 Requerimiento

Después del incidente del registro duplicado, servicios escolares pide rastrear los cambios: "Cada vez que se actualice un registro de la tabla alumnos, debe quedar constancia automática en una bitácora de auditoría, sin que ninguna persona tenga que acordarse de registrarlo." Ese es exactamente el trabajo de un disparador.

La tabla de auditoría ya está creada

🗄 SistemaAdministracionEstudiantil › auditoria_alumnos
id_eventomatriculafecha_eventodescripcion
1231004592026-05-28 10:14:02Registro actualizado

Tu disparador debe insertar una fila como esa cada vez que ocurra el evento. id_evento se genera solo (es autoincremental); tú solo insertas las otras tres columnas.

El simulador usa SQLite: 3 pequeñas diferencias con MySQL

1) Para fechas usa datetime('now') en lugar de NOW(). 2) El autoincremental se escribe AUTOINCREMENT (sin guion bajo). 3) Los disparadores envuelven su acción entre BEGIN y END;. Todo lo demás de este cuaderno (SELECT, WHERE, INSERT, UPDATE, DELETE, INNER JOIN) funciona idéntico.
