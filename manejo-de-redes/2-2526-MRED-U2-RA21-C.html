C-01

¿Qué significa verificar la conectividad?

Verificar la conectividad significa comprobar, paso a paso, que los dispositivos de una red realmente pueden comunicarse entre sí. No basta con conectar cables y asignar direcciones IP: también debemos revisar interfaces, gateway, respuestas de ping y rutas.

Explicación clara

Imagina que una red es como una colonia. Cada casa tiene dirección, pero para saber si puedes llegar a otra casa necesitas revisar la calle, la salida principal, el camino y si la otra persona responde. En redes, esa revisión se hace con comandos como ping, show ip interface brief y tracert/traceroute.

Elemento a revisarPregunta que responde
Stack TCP/IP local¿Mi propia computadora o router tiene funcionando la red?
Interfaz¿El puerto está activo, con IP y en estado correcto?
Gateway¿El equipo puede salir de su red local?
Host remoto¿Puedo comunicarme con otro dispositivo fuera de mi red?
Ruta¿Por qué camino viajan los paquetes?
C-02

Prueba de stack: comando ping y loopback

La primera prueba sencilla es revisar si el dispositivo puede comunicarse consigo mismo. Para eso se usa la dirección de loopback 127.0.0.1 en una PC o comandos de ping desde el router.

Idea clave

Si un equipo no puede hacer ping a sí mismo, el problema puede estar en su configuración TCP/IP o en el sistema local. Si sí responde, el siguiente paso es probar hacia otros dispositivos.

PC – Prueba de loopback
C:\> ping 127.0.0.1

Respuesta desde 127.0.0.1: bytes=32 tiempo<1ms TTL=128

C-03

Verificación de interfaces del router

Antes de culpar a la red, se revisa si las interfaces están activas, tienen dirección IP y se encuentran en estado correcto. El comando más usado para una revisión rápida es:

Router – Resumen de interfaces
Router# show ip interface brief

Interface              IP-Address      OK? Method Status                Protocol
GigabitEthernet0/0     192.168.1.1     YES manual up                    up
GigabitEthernet0/1     unassigned      YES unset  administratively down down
Serial0/0/0            10.0.0.1        YES manual up                    up
Cómo interpretar estados de interfaz
up / up
La interfaz está encendida y el protocolo de línea funciona. Es el estado deseado.
administratively down / down
La interfaz está apagada por configuración. Normalmente se corrige con no shutdown.
up / down
La interfaz está activa físicamente, pero el protocolo no funciona. Puede ser cable, encapsulación, velocidad o configuración.
down / down
No hay conexión física o el puerto está apagado/con problema.

Prueba de conectividad del router, switch y red local

Una verificación ordenada ayuda a encontrar más rápido el error. Se recomienda probar de lo más cercano a lo más lejano.

Consejo de laboratorio

En Packet Tracer, si el ping falla la primera vez, inténtalo una segunda vez. A veces el primer intento se usa para resolver ARP y los siguientes ya responden correctamente.

C-05

Gateway, hosts remotos y rastreo de ruta

El gateway es la salida de una red local hacia otras redes. Si una PC puede hacer ping a equipos de su misma red, pero no puede llegar a redes externas, se debe revisar el gateway y las rutas.

PC – Prueba de gateway y host remoto
C:\> ping 192.168.1.1   # Gateway
Respuesta desde 192.168.1.1

C:\> ping 10.0.0.20      # Host remoto
Tiempo de espera agotado

Cuando el ping a un destino remoto falla, se puede usar tracert en Windows o traceroute en IOS para observar por dónde intenta viajar el paquete.

Rastreo de ruta
C:\> tracert 10.0.0.20

  1     <1 ms     1 ms     1 ms  192.168.1.1
  2      *        *        *     Tiempo de espera agotado.
Interpretación rápida

Si el salto 1 responde, la PC llega al gateway. Si después se pierde, el problema puede estar en el router siguiente, en una ruta faltante, en una interfaz apagada o en una red destino mal configurada.
